The Concept

What is (De)Buggy Board?

An interactive, educational breadboard to ease circuit-building and debugging.

(De)Buggy Board is an interactive, educational breadboard comprising of hardware and software components. Students design the circuit they want to build on a software user interface. Then, as they build it on the breadboard, their circuit design will sync with the software to provide real-time information about their circuit.

The Motivation

The future of introductory circuits education.

This is (De)Buggy Board.

With traditional breadboards, designs can quickly become a dense forest of wires. This requires tedious testing strategies that offer little educational value and tend to be time consuming.

With the (De)Buggy Board, the student will be able to see what voltages are at each node to quickly notice when something simple goes awry. This allows the student to quickly prototype the circuit and get started with the lab at hand.

The Requirements

Hardware

Functional

Nodal Voltages

Measure all DC nodal voltages on breadboard and relay voltage data to software.

Auto-Routing

Automatically connect pin rows on the breadboard without wires.

Status LED's

Indicate the status of each pin row on the breadboard via an RGB LED.

Software

Functional

Circuit Diagramming

Provide a user interface for symbolically diagramming circuits. Automatically perform basic error checks on the circuit diagram via KCL and KVL.

Placing and Wiring Components

Create an interface for placing and wiring circuit components from the circuit diagram on on virtual breadboard.

Debugging Feedback

Provide real-time debugging feedback derived from nodal voltages. Is my resistor disconnected? Did I place the capacitor in the correct pins?

Auto-Routing Circuit Visual

Allow for the user to see how their circuit is changed when using auto-routing due to internal resistance of routing fabric (See Components).

Non-Functional

Safety

The circuit should detect dangerously invalid circuits (such as short circuits and high currents) and avoid unsafe operating conditions by shutting down. As first priority, the circuit should not pose physical harm to users through unsafe operation. Second, the circuitâ€™s safety system should prevent internal damage.

Reliability

We will have reliable and accurate DC nodal voltage readings for students to successfully understand the physical circuit they are prototyping.

The Components

What we need to build this.

A.K.A our parts list.

Crosspoint Switch

Required to make auto-routing possible. The main design choice here was buffered or unbuffered. Buffered has low internal resistance but has amplifiers on the outputs, which means we are "imitating" the user's signals and not routing them. This can cause problems in circuits that use feedback. The unbuffered switch just contains a matrix of analog switches. This means a higher internal resistance (approx 200 ohms), but now we actually route signals instead of recreating them. They are also cheaper!

Analog to Digital Converter

To sense the DC voltages, we need an external ADC. We cannot use the MCU's on-chip ADC since the voltage reference of the MCU's ADC is limited by the max supply voltage of the MCU. Limiting nodal voltage sampling to 0 to 5v is not user-friendly. Therefore, our ADC will have high supply rails (+/- 12v) to give the user more possibilities.

Microcontroller

We need only a simple MCU. We decided to pick one with USB support in order to make our communication protocol with the software application simple. Atmel MCU hack support is fantastic online since the processor is widely used in arduino chips. This MCU is responsible for receiving commands from the application to control the RGB LEDs, the crosspoint switches, and track nodal voltages.

Analog Mux

Used to multiplex in all the breadboard nodes to the external ADC for DC voltage sampling. These muxes also need to have the high supply rails of our ADC (+/- 12v) in order to allow for the signals to pass through correctly.

Shift Registers

Used to minimize IO from the MCU used to control the RGB LEDS along the breadboard. Nothing crazy here. Just some simple 8 bit shift registers. We need 6 ICs per a given color to control a 48 node breadboard with an RGB LED for each node. Therefore, we need 18 total ICs.

Small Breadboard

The all-important breadboard. Due to our budget, we can only afford a 48 node board. We achieve this by daisy chaining crosspoint switches. Using a 16x16 switch, we need a grid of 9 switches total to have 48 unique nodes. These 9 switches will exhaust the majority of our budget.

RGB LEDs

The SMD RGB LEDs used for each pin row on the breadboard (48 total).

Competitive Analysis

Egyptian Team

An eight-person team from Egypt designed an auto routing breadboard that uses a software backend to route nodes. Their design and goals are similar to ours, but lack the real-time voltage tracking that is essential for our debugging application.

Learn More

No Wires Breadboard

This the only product similar to our concept that we could find. It has features similar to ours, such as an attached software to plan your breadboard wiring, and LEDs to guide placement. As it turns out, this product is actually made by the Egyptian Team. This product is currently not for sale as it is still in development.

Learn More

Build18 Projects

There have been two similar projects within Carnegie Mellonâ€™s annual Build18 hardware hackathon.

The ProtoSmart team created a breadboard that sensed voltages at each row of a small breadboard. However, that design was limited by the on-chip analog to digital converters of the microcontroller which can usually only sense 0 to 5v signals.

The Field Programmable Protoboard team set out to make a breadboard that wires rows together for you (auto-routing). However, the sheer size of the routing became to complex to accomplish for them using only transistors and shift registers. This design was also only limited to routing digital circuits.

Use Cases

Firstly, this is what an ideal use case will look like:

  • User inputs symbolic circuit schematic into the circuit diagrammer tool and notifies system when circuit is finished.
  • Software then gives feedback that the circuit is complete (in that all components are wired together).
  • User clicks components from the schematic and then places them onto the virtual breadboard diagrammer until all components are placed.
  • Software then gives feedback that both the circuit is complete and the symbolic diagram matches the breadboard diagram.
  • User places the components on the real breadboard.
  • User clicks "Run."
  • Hardware scans real breadboard nodal voltages and sends the data to the software.
  • Software displays nodal board voltages on the virtual breadboard and circuit diagram.
  • Software compares nodal voltage information to theoretical voltage analysis and verifies that voltages are in the right range of values.
  • User uses the nodal voltage information to gauge that the circuit was created correctly.

But of course, since this is a *debugging* board, error use cases that could come up are:

  • Software gives feedback that the symbolic circuit is incomplete (not all components are wired together), and highlights with a bolded dot where certain components are not connected.
  • Software gives feedback that the symbolic diagram does not match the breadboard diagram, and highlights via bolded nodes which nodes have no pair node in the other diagram.
  • Software compares nodal voltage information to theoretical voltage analysis, sees that the values differ from what they should be, and red rows in the software light up and the information is sent to the hardware.
  • Hardware has red LEDs on the physical breadboard light up where nodes have incorrect voltages.
  • User uses the nodal voltage information to gauge that the circuit was not created correctly as compared to their own theoretical expectations.
